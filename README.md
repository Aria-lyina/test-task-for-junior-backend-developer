# Task Service

Сервис для управления задачами с HTTP API на Go.

## Требования

- Go `1.23+`
- Docker и Docker Compose

## Быстрый запуск через Docker Compose

```bash
docker compose up --build
```

После запуска сервис будет доступен по адресу `http://localhost:8080`.

Если `postgres` уже запускался ранее со старой схемой, пересоздай volume:

```bash
docker compose down -v
docker compose up --build
```

Причина в том, что SQL-файл из `migrations/0001_create_tasks.up.sql` монтируется в `docker-entrypoint-initdb.d` и применяется только при инициализации пустого data volume.

### Правила периодичности при создании шаблона

При создании `repeat_task` нужно задать **ровно одно** из трёх:

- `period_id` — стандартный период из справочника, RRULE берётся автоматически
- `rrule` — произвольное iCalendar RRULE-правило, например `FREQ=DAILY;INTERVAL=10`
- `custom_dates` — массив конкретных дат в формате `YYYY-MM-DD`

Пример создания шаблона с периодом из справочника:
```json
{
  "title": "Ежедневный обзвон пациентов",
  "description": "Обзвонить пациентов из списка ожидания",
  "status": "new",
  "period_id": 1,
  "enabled": true
}
```

Пример с кастомным RRULE (каждые 10 дней):
```json
{
  "title": "Резервное копирование",
  "rrule": "FREQ=DAILY;INTERVAL=10",
  "enabled": true
}
```

Пример с конкретными датами:
```json
{
  "title": "Праздничные задачи",
  "custom_dates": ["2026-01-01", "2026-02-23", "2026-03-08", "2026-05-01"],
  "enabled": true
}
```

Обоснование выбора таблиц и архитектуры

Я выбрала создать таблицы период и периодические задачи так как считаю что для задач не должно быть разделения в общем списке, задача созданная через повторение будет отмечена за число какое-то и висеть в общем листе задач. То есть изменение параметров повторной задачи не должно влиять на дочерние задачи если нет конкретного намерения их поменять (хотя я считаю что историю задач вообще менять не стоит) Справочник создан для удобства подстановки правил, но у каждой задачи правило так же задано, чтоб не запрашивать каждый раз по id период, и была возможность задавать его вручную. Выбан тип rrule так он стандарт, запросы на бек без фронта не удобно вручную его писать, но на фронте будет легко его создать для отправки на бек, есть специальные компоненты. ### RRULE резолвится в сервисе, а не в репозитории

При создании шаблона с `period_id` сервис сразу запрашивает `rrule_template` из периода и сохраняет итоговое правило в поле `rrule` самого шаблона. Репозиторий просто сохраняет то что получил — он не знает о логике резолвинга.

Это даёт два преимущества: во-первых, генерация задач не делает лишний JOIN к таблице `periods` каждый раз. Во-вторых, если период в справочнике будет изменён или удалён, уже созданные шаблоны продолжат работать со своим правилом.

### Генерация задач через cron

Сервис запускает cron-планировщик (библиотека `robfig/cron`) внутри процесса. Каждый день в 00:00 по московскому времени вызывается `GenerateAllTasks` — метод обходит все активные шаблоны и создаёт задачи на текущий день.

Дополнительно есть эндпоинт `GET /api/v1/tasks-generate` для ручного запуска генерации и `POST /api/v1/repeat-tasks/{id}/generate` для генерации по конкретному шаблону в произвольном диапазоне дат.

Поле `last_generated_at` в шаблоне хранит дату последней генерации — повторная генерация за уже покрытый период не создаст дублей.
